## Tasks

### Phase 1: Core data structures and single-symbol matching
1. Set up the CMake skeleton, GoogleTest integration via FetchContent, and a passing "hello world" build per target.
2. Implement `OrderPool` with the debug-mode allocation tripwire.
3. Build `apps/bench/main.cpp` skeleton (synthetic event generator, latency histogram via HDRHistogram).

### Phase 4: Seqlock-protected top-of-book and sampler
1. Implement the sampler thread on its own pinned core. 30 Hz polling cadence; build a JSON delta diff against the prior published snapshot.
2. Wire spdlog with level filtering: `SPDLOG_TRACE` calls compile to nothing in release.
3. The seqlock protocol implementation is owned by the Market Microstructure Engineer in coordination with the Concurrency Reviewer; this agent integrates the resulting `TopOfBookSnapshot` API into the sampler thread but does not invent the protocol.

### Phase 5: Benchmark hits 6M events per second
1. Add PGO support to the CMake config (instrumented build, training run, optimized rebuild).
2. Wire `bench/baseline.json` regeneration into the CI workflow's manual trigger.
3. Profile-driven cleanup of any allocations or atomic costs the Performance Engineer flags.

### Phase 6: NASDAQ ITCH 5.0 replay
1. Implement `include/meridian/itch.hpp` as a single-header parser. Cover Add Order, Add Order with MPID, Order Executed, Order Executed With Price, Order Cancel, Order Delete, Order Replace, Trade Non-Cross, Cross Trade, Trading Action, Stock Directory.
2. Implement `apps/replay/main.cpp`: open the tape, parse messages, dispatch to the matching engine, emit JSON Lines on stdout.
3. Verify against a small public ITCH sample under `tests/integration/`.

### Phase 8: WebSocket server and protocol
1. Wire uWebSockets via FetchContent.
2. Implement `apps/server/main.cpp`: start replayer, matching loop, sampler, and uWebSockets event loop on pinned cores per the design spec.
3. Implement the WebSocket protocol: on connect, send `snapshot` with L10 book + recent trades + perf metrics; thereafter the sampler builds and broadcasts `delta` messages at 30 Hz.
4. Document the protocol at `docs/api/websocket.md`.

### Phase 10: Hosting and CI/CD
1. Pair with the DevOps Engineer on the systemd service unit, `meridian.toml` production config, and log rotation.
2. Make sure `meridian-server --help` shows every flag with examples.

* ThreadSanitizer is clean for the seqlock paths.
* The ITCH parser correctly handles a public sample tape; the integration test passes.
* The WebSocket server holds 30 Hz cadence under a 100-client load test.
